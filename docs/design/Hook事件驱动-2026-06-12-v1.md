# Hook 事件驱动

> 版本: v1 | 日期: 2026-06-12 | 作者: hongchuwudi

## 要解决的问题

一次 tick 从取行情到存数据库有 8 个步骤。不能写成一长串 if-else，也不能让上游关心下游的存在。需要一条**自动级联的流水线**——上游产出数据，下游自动响应。

## 设计思路

不定义抽象的 Hook 注册表，而是把钩子硬编码在 `_tick()` 的顺序里。每个步骤的输出自动触发下一个步骤：

```
行情拉到 ──→ 5 Agent 决策
              │
              ├─ 调度师完成 ──→ 分析师 ∥ 复盘师 并行启动
              ├─ 分析完成    ──→ 风控师审查
              ├─ 风控不通过  ──→ 退回分析师重做（≤2次）
              ├─ 风控通过    ──→ 裁决员综合
              └─ 裁决完成    ──→ 返回决策 dict

决策生成   ──→ RiskService 检查 ──→ 不通过则拦截
交易执行   ──→ 平仓时自动回写盈亏到记忆库
持仓管理   ──→ 每 tick 无条件更新追踪止损（含 HOLD 时）
网络故障   ──→ 代理自动切换
连续失败   ──→ 熔断触发
tick 完成  ──→ 持久化 + 缓存刷新 + WebSocket 推送 + 学习反馈写入
```

**不抽象的原因**：当前只有一条流水线，抽象成事件总线只会增加复杂度。未来如果需要动态编排（非固定顺序），再迁移到真正的 EventBus 模式。

## 各环节详解

### Hook 1: 行情 → 决策

```python
# _tick() → loop.py:118-132
# 行情是所有决策的前提。拉不到行情，后续全不执行。
df = await self.market_data.get_ohlcv(...)
price = await self.market_data.get_current_price(...)
# ↓ 触发
decision = await self.coordinator.analyze(df, price, equity, position)
```

### Hook 2: Agent 间级联（5 步流水线）

```python
# coordinator.py:84-159

# Step 1: 调度师 → 输出 focus/priority
sch = await scheduler.invoke(...)
# ↓ 触发 Step 2
analyst, reviewer = await asyncio.gather(analyst.invoke, reviewer.invoke)
# ↓ 触发 Step 3
risk = await risk.invoke(...)
# ↓ 风控检测到问题 → 触发回退
while risk.handoff == 'analyst' and redo < 2:
    analyst = await analyst.invoke(...)  # 重新分析
    risk = await risk.invoke(...)        # 重新审查
# ↓ 触发 Step 4
trader = await trader.invoke(...)
# ↓ 输出最终决策
return {"signal": "BUY", "stop_loss": 0.12, ...}
```

### Hook 3: 决策 → 风控拦截

```python
# loop.py:187-198
risk_result = await self.risk.check(signal, equity, position_value)
if not risk_result.passed:
    # 拦截！不执行交易，但正常持久化（记录被拦截的决策）
    await persist_tick(...)
    return  # 终止本 tick
```

风控检查链（5 项级联，risk.py:47-57）：
```
熔断开关 → 日回撤 → 日亏损 → 仓位上限 → 开仓频率
  任一失败 → 立即返回拦截，后续检查不再执行
```

### Hook 4: 平仓 → 学习反馈

```python
# loop.py:210-219
if trade_result.action in ("close", "reverse"):
    memory_service.update_outcome(memory_id, pnl)
# 把盈亏写回开仓时的决策记录，供复盘师检索学习
```

### Hook 5: 网络故障 → 代理切换

```python
# loop.py:119-132
for attempt in range(2):
    try:
        df = await fetch_ohlcv(...)
        break
    except Exception:
        if attempt == 0 and self.exchange.switch_to_backup():
            continue  # 切到备用代理，重试一次
        raise

# 下个 tick 开始时会先试切回主代理（loop.py:119-120）
if self.exchange.use_backup:
    self.exchange.switch_to_primary()  # 试主代理
```

### Hook 6: 连续失败 → 熔断

```python
# loop.py:239-249
except Exception:
    self._fail_count += 1
    if self._fail_count >= 10:
        await self.risk.trip_circuit_breaker(
            f"连续 {self._fail_count} 次 tick 失败"
        )
else:
    self._fail_count = 0  # 成功一次就清零
```

### Hook 7: tick 完成 → 持久化 + 推送

```python
# loop.py:221-236
await persist_tick(...)       # PostgreSQL
await cache_signal(signal)    # Redis 缓存刷新
await publish_event({         # Redis Pub/Sub → WebSocket → 前端
    "type": "tick_complete",
    "signal": signal.signal,
    ...
})
```

## 为什么不用抽象的事件总线

| 抽象总线 | 当前硬编码 |
|---------|----------|
| 需要定义 Event 类型、Listener 注册、调度器 | 零额外代码 |
| 事件触发顺序不可见（需看注册顺序） | 代码从上往下读就是执行顺序 |
| 调试需追踪多个注册点 | 一个文件看全局 |
| 适合多发射源、多消费者 | 当前只有一个发射源（tick） |
