# 定时调度

> 版本: v1 | 日期: 2026-06-12 | 作者: hongchuwudi

## 要解决的问题

7×24 无人值守交易，每 6 分钟执行一次决策循环。需要一个精确、可靠、可多实例部署的定时触发机制。

## 为什么这样设计

### 1. 对齐整倍数时间点，而非简单累加

```python
# 不好：阻塞式 sleep + 累加间隔
time.sleep(360)  # 阻塞整个线程，6 分钟内什么都不能干
next = now + 360 # 一次 tick 执行 10 秒，下次就比预期晚 10 秒
                  # 日积月累严重漂移：第一天 00:00，第七天 23:30

# 好：非阻塞 asyncio.sleep + 对齐 epoch 整倍数
next_ts = ((time.time() // 360) + 1) * 360  # 永远是 xx:06:00, xx:12:00...
await asyncio.sleep(next_ts - time.time())  # 交控制权给事件循环
```

**`asyncio.sleep` 和 `time.sleep` 的关键区别**：

| | `time.sleep(360)` | `await asyncio.sleep(360)` |
|---|---|---|
| 线程 | 阻塞整条线程 | 不阻塞，交出控制权 |
| 6 分钟内 | 什么都干不了 | 还能处理 WebSocket、API 请求 |
| 适用场景 | 脚本 | 同时跑 API + WebSocket + 引擎的服务 |

**为什么对齐 epoch**：
- 无论单次 tick 执行多久（40 秒还是 200 秒），下一个触发时间永远在整倍数点。
- 日志可追溯——你永远知道 14:30:00 有一次 tick，而不是"大概 14:30 左右"。

### 2. Redis 分布式锁防重执行

```python
# 用 SET NX EX 原子操作抢锁
acquired = redis.set("lock:scheduler:tick", "1", nx=True, ex=interval)
```

**原因**：
- 未来可能多实例部署（主备切换），两个实例同时触发 tick 会导致重复下单。
- `SET NX EX` 是原子操作：获取锁 + 设置过期时间一步完成，不会死锁。
- 锁过期时间 = tick 间隔。即使进程崩溃未释放锁，下个周期自动过期。

### 3. 启动时清理残留锁

```python
redis.delete("lock:scheduler:tick")
```

**原因**：异常关闭时锁未释放，重启后第一个 tick 直接抢到锁，不用等过期。

## 代码实现

```python
# scheduler.py

class SchedulerService:
    def __init__(self, interval_seconds=360, align_to_quarter=True):
        self._interval = interval_seconds

    async def run_loop(self, callback):
        redis = await get_redis()
        await redis.delete("lock:scheduler:tick")  # 清理残留锁

        while self._running:
            await self._wait_for_next()             # 等到达整倍数时间点
            if not await self._acquire_lock():      # 抢分布式锁
                continue                            # 抢不到跳过
            try:
                await callback()                    # 执行 tick
            finally:
                await self._release_lock()          # 释放锁

    async def _wait_for_next(self):
        now = time.time()
        # 对齐到 interval 整倍数
        next_ts = ((now // self._interval) + 1) * self._interval
        sleep_s = next_ts - now
        if sleep_s > 0:
            # asyncio.sleep 交出控制权，不阻塞事件循环
            # 等待期间仍可处理 WebSocket / HTTP 请求
            await asyncio.sleep(sleep_s)

    async def _acquire_lock(self):
        return bool(await redis.set(
            "lock:scheduler:tick", "1", nx=True, ex=self._interval
        ))
```

## 调用链

```
run_loop(_tick)  ─── 每 6 分钟调用一次 _tick()
  │
  └─ _wait_for_next()  ── asyncio.sleep 到 00:06:00
  └─ _acquire_lock()    ── Redis SETNX 抢锁
  └─ _tick()            ── 完整交易周期（约 40-200 秒）
  └─ _release_lock()    ── 释放锁
  └─ 循环
```

## 关键权衡

| 选择 | 放弃 |
|------|------|
| asyncio.sleep 对齐 epoch | 无法精确到毫秒（不需要） |
| Redis 分布式锁 | 额外依赖 Redis，但换来多实例安全 |
| 抢不到锁直接跳过 | 不会排队等待（下一轮自然会执行） |
| 锁过期 = tick 间隔 | 进程崩溃后最多损失一个周期 |
